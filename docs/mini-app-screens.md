# Экраны Telegram Mini App

Документ описывает MVP-экраны и основные состояния UI. Это не дизайн-макет, а
карта продукта для frontend-разработки.

## Общие принципы UI

- Везде использовать термин `семья`.
- Не использовать `club` в интерфейсе.
- Не показывать платежные реквизиты до подтверждения доступа участником.
- Не показывать имена, фото и профили участников в публичной карточке семьи.
- Показывать владельца в счетчике участников.
- Все критичные действия должны иметь понятный статус и deep link из Telegram.
- Интерфейс должен выглядеть как приложение внутри Telegram, а не как обычный
  сайт: нижняя навигация, мобильная плотность, быстрые состояния.
- Использовать Telegram theme params, safe-area padding и системную BackButton
  для внутренних экранов.
- Использовать haptic feedback только на важных действиях, чтобы не перегружать
  устройство пользователя.

## 1. Username Gate

Показывается, если у Telegram-пользователя нет username.

Содержимое:

- объяснение, что без username нельзя использовать SubsMarket;
- инструкция открыть настройки Telegram и создать username;
- кнопка `Я создал username`;
- кнопка повторной проверки.

Дальше пользователь не проходит, пока `GET /api/me` возвращает
`USERNAME_REQUIRED`.

## 2. Главная

Блоки:

- поиск семей;
- мои заявки;
- мои семьи;
- платежи;
- создать семью;
- уведомления.

На первом этапе показываются семейные подписки и отдельный раздел семейных
тарифов. Аккаунты и ГБ можно показать позже как Marketplace Engine, но не
смешивать с Family.

## 3. Каталог сервисов

Назначение:

- выбрать сервис для поиска или создания семьи.

Элементы:

- категории;
- поиск по названию;
- карточка сервиса;
- статус сервиса.

Показывать только активные сервисы. `pending_verification` не доступен для
создания семьи.

## 4. Поиск семей

Карточка семьи до заявки показывает:

- сервис;
- период оплаты;
- общую стоимость;
- долю участника;
- счетчик `3 из 6`;
- свободные места;
- дату следующей оплаты;
- описание;
- дату создания;
- имя и фото владельца;
- кнопку `Отправить заявку`.

Не показывает:

- username владельца;
- платежные реквизиты;
- участников;
- Telegram-профили участников.

Если пользователь уже получил отказ в этой семье, семья не показывается в
поиске.

Если семья заполнена, она не показывается в поиске.

## 5. Карточка семьи

### До заявки

Действие:

- `Отправить заявку`.

Проверки:

- нет pending заявки в эту семью;
- нет отказа в этой семье;
- не больше трех pending заявок на этот сервис.

При четвертой активной заявке:

`У вас уже 3 активные заявки на {service}. Дождитесь ответа или отмените одну заявку.`

### После заявки

Показывать:

- статус `Ожидает ответа владельца`;
- username владельца;
- кнопку открыть Telegram-чат с владельцем;
- кнопку `Отменить заявку`.

### После отказа

Карточка больше не показывается в поиске. В истории заявок статус:

- `Отклонена`;
- текст `Заявка отклонена. Вы можете отправить заявку в другую семью.`

### Истекла

Статус:

- `Истекла`.

Действие:

- можно отправить заявку снова;
- можно выбрать другую семью.

## 6. Создание семьи

Шаги:

1. Выбор сервиса.
2. Выбор периода: месяц или год.
3. Выбор вместимости.
4. Ввод общей стоимости подписки.
5. Автоматический расчет доли и округления.
6. Выбор даты оплаты.
7. Описание.
8. Правила владельца.
9. Платежный реквизит: банк и номер телефона.
10. Предпросмотр карточки.
11. Создать.

Валидации:

- вместимость не выше лимита сервиса и 8;
- владелец входит в вместимость;
- банк только Kaspi, Halyk, Freedom, Jusan;
- только номер телефона, не карта и не IBAN;
- максимум две активные семьи владельца.

## 7. Мои заявки

Показывает заявки пользователя.

Статусы:

- `Ожидает ответа владельца`;
- `Принята`;
- `Отклонена`;
- `Истекла`;
- `Отменена`;
- `Семья заполнена`.

Действия:

- открыть семью;
- отменить pending заявку;
- отправить новую заявку, если правила разрешают.

## 8. Заявки владельца

Показывает pending заявки старые сверху.

Карточка заявки:

- имя кандидата;
- фото;
- username;
- дата заявки;
- кнопка `Принять`;
- кнопка `Отклонить`.

Отклонение без причины. Поле причины не показывать.

После принятия:

- создается участник `awaiting_access`;
- владелец видит задачу `Выдать доступ`.

## 9. Участник ожидает доступ

Состояние участника:

- `awaiting_access`.

Показывать участнику:

- владелец принял заявку;
- реквизиты еще скрыты;
- деньги переводить не нужно;
- можно открыть чат с владельцем;
- можно отменить вступление.

Показывать владельцу:

- участник ждет доступа;
- кнопка `Доступ выдан`;
- кнопка `Отменить вступление`.

Если владелец отменил, кандидат получает уведомление.

## 10. Подтверждение доступа

После `Доступ выдан` участник видит:

- инструкция проверить подписку;
- кнопка `Доступ получил`;
- кнопка открыть чат с владельцем;
- предупреждение, что через 24 часа придет напоминание.

Если нет подтверждения за 24 часа:

- участник и владелец получают напоминание;
- место остается занятым;
- платеж не создается.

Владелец видит кнопку `Напомнить подтвердить` и может отправить повторное
уведомление. Автоматического удаления в этом сценарии нет.

## 11. Первый платеж

После `Доступ получил`:

- открыть реквизиты;
- показать сумму;
- показать 30-минутный таймер;
- кнопка `Оплатил`;
- кнопка открыть чат с владельцем.

После 30 минут без `Оплатил`:

`Время на первый платеж истекло. Если вы уже оплатили, нажмите "Оплатил". Если нет - оплатите или напишите владельцу.`

Просрочка не удаляет участника автоматически.

## 12. Подтверждение оплаты владельцем

Владелец видит платеж `Оплатил`.

Действия:

- `Подтвердить`;
- `Не получил оплату`.

При `Подтвердить`:

- платеж `paid`;
- участнику уведомление `Владелец подтвердил оплату.`

При `Не получил оплату`:

- платеж возвращается в `due`;
- участнику уведомление:
  `Владелец не подтвердил получение оплаты. Проверьте перевод или свяжитесь с владельцем.`

## 13. Регулярные платежи

Участник видит:

- ближайший платеж;
- сумму;
- дату;
- статус.

Напоминания:

- месячный период: за 3 дня, в дату оплаты, ежедневно после даты до `Оплатил`;
- годовой период: за месяц, за 3 дня, в дату оплаты, ежедневно после даты до
  `Оплатил`.

После `Оплатил` напоминания участнику прекращаются, владелец подтверждает
получение.

## 14. Управление семьей владельца

Блоки:

- состав семьи;
- ожидающие заявки;
- участники без доступа;
- платежи на подтверждение;
- просрочки;
- настройки описания;
- изменение цены;
- закрытие семьи.

Нельзя менять:

- сервис;
- вместимость;
- период;
- правила владельца.

## 15. Удаление участника

Владелец нажимает `Удалить`.

Перед удалением владелец выбирает причину:

- не оплатил;
- нет связи;
- проблема с доступом;
- по договоренности;
- другое.

Правила:

- удаление выполняется сразу;
- место освобождается сразу;
- будущие платежи отменяются;
- участник получает уведомление;
- причина и действие сохраняются в истории.

## 16. Закрытие семьи

Владелец запускает закрытие.

Участники получают:

- дата закрытия;
- кнопка подтверждения понимания.

Правила:

- предупреждение 3 календарных дня;
- новые заявки не принимаются;
- отсутствие подтверждения не продлевает закрытие;
- будущие платежи отменяются;
- история остается.

## 17. Уведомления

Минимальные уведомления MVP:

- заявка создана;
- заявка отменена кандидатом;
- заявка отклонена;
- заявка истекла;
- семья заполнена для pending кандидата;
- заявка принята;
- владелец отменил вступление до доступа;
- доступ выдан;
- доступ не подтвержден за 24 часа;
- владелец повторно напомнил подтвердить доступ;
- открыт первый платеж;
- первый платеж просрочен;
- участник нажал `Оплатил`;
- владелец подтвердил оплату;
- владелец нажал `Не получил оплату`;
- регулярный платеж скоро;
- удаление участника;
- закрытие семьи.
