# План разработки

Цель: собрать новый SubsMarket поэтапно, без переноса старого `Club`-мышления.

## Правила работы

- Сначала реализуется Family Engine: `Семья подписки` и отдельная
  `Семья тарифа`.
- Не добавлять аккаунты, ГБ, споры и рейтинг в первый релиз.
- Не добавлять `slot_config`, `slot_type`.
- Не делать универсальную сущность для семей и объявлений.
- Не хранить телефон пользователя.
- Не показывать платежные реквизиты до подтверждения доступа участником.
- Учитывать Telegram Mini App: `initData`, theme params, safe-area, BackButton и
  haptic feedback.

## Этап 0. Подготовка проекта

Результат:

- backend scaffold;
- frontend Telegram Mini App scaffold;
- Postgres;
- миграции;
- линтеры;
- генерация OpenAPI/типов;
- базовый CI или локальные команды проверки.

Критерий готовности:

- пустое приложение открывается;
- backend healthcheck работает;
- миграции применяются на чистую БД.

## Этап 1. Identity и Catalog

Функции:

- проверка Telegram init data;
- создание/обновление `User`;
- блокировка функционала без username;
- импорт `family-services.json`;
- список активных сервисов.

Критерий готовности:

- пользователь без username видит gate;
- пользователь с username видит главную;
- каталог отдает только active сервисы.

## Этап 2. Создание и поиск семьи

Функции:

- создать семью подписки или семью тарифа;
- расчет доли с округлением вверх до 50 KZT;
- лимит 8 участников;
- владелец входит в счетчик;
- максимум две активные семьи владельца;
- платежный реквизит владельца шифруется;
- публичная карточка семьи;
- поиск видимых семей.

Критерий готовности:

- владелец создает семью;
- другой пользователь видит карточку без username и реквизитов;
- заполненные и closing семьи не видны в поиске.

## Этап 3. Заявки

Функции:

- отправить заявку;
- открыть username владельца после заявки;
- владелец видит кандидата;
- очередь заявок старые сверху;
- принять;
- отклонить без причины;
- отменить кандидатом;
- истечение через 24 часа;
- family_full auto-cancel;
- лимит 3 active заявок на сервис;
- скрытие семьи после отказа.

Критерий готовности:

- нельзя отправить 4-ю active заявку на один сервис;
- после отказа семья скрыта для кандидата;
- после `expired` можно снова подать заявку;
- уведомления создаются.

## Этап 4. Доступ перед оплатой

Функции:

- после принятия создается `FamilyMember(awaiting_access)`;
- отмена вступления обеими сторонами до выдачи доступа;
- владелец отмечает доступ выдан;
- участник подтверждает доступ;
- напоминание через 24 часа без автоматического удаления;
- владелец может повторно отправлять напоминание до подтверждения;
- открытие реквизитов только после подтверждения доступа.

Критерий готовности:

- платеж не создается до подтверждения доступа;
- реквизиты не возвращаются до подтверждения доступа;
- если доступ не подтвержден за 24 часа, место остается занятым;
- удалить участника может только владелец по обычному правилу с предупреждением.

## Этап 5. Первый платеж

Функции:

- создать `FamilyPayment(first, due)`;
- 30-минутный срок;
- `Оплатил`;
- отмена `Оплатил`;
- подтверждение владельцем;
- `Не получил оплату`;
- overdue без auto-remove.

Критерий готовности:

- первый платеж переходит по всем статусам;
- после подтверждения участник становится `active`;
- просрочка не удаляет участника;
- уведомления соответствуют текстам из спецификации.

## Этап 6. Регулярные платежи

Функции:

- создание scheduled платежей;
- месячные напоминания: за 3 дня, в дату, ежедневно после;
- годовые напоминания: за месяц, за 3 дня, в дату, ежедневно после;
- overdue через 24 часа после даты;
- предоплата одного будущего периода;
- несколько периодов вручную владельцем.

Критерий готовности:

- регулярный платеж не использует 30-минутный срок;
- подтвержденная предоплата отключает напоминания;
- изменение цены не меняет уже созданные платежи.

## Этап 7. Управление семьей

Функции:

- редактирование описания;
- изменение цены раз в месяц;
- изменение даты оплаты до первого полного заполнения;
- добровольный выход участника;
- немедленное удаление владельцем с обязательной причиной;
- закрытие семьи с 3-дневным предупреждением.

Критерий готовности:

- участник выходит и место освобождается;
- удаление сразу освобождает место и сохраняет причину в истории;
- закрытие отменяет будущие платежи и скрывает семью из поиска.

## Этап 8. Audit и hardening

Функции:

- `FamilyAuditLog` для всех критичных действий;
- защита гонок при одобрении последнего места;
- защита гонок при лимитах заявок;
- тесты доменных правил;
- интеграционные тесты API.

Критерий готовности:

- два кандидата не могут занять одно последнее место гонкой;
- audit покрывает заявки, доступ, платежи, удаление и закрытие;
- основные сценарии проходят автотестами.

## Минимальный тестовый набор

### Unit tests

- расчет доли и округления;
- запрет изменения сервиса, вместимости и периода;
- request state machine;
- member state machine;
- payment state machine;
- price change rules;
- due date fallback for 29/30/31.

### Integration tests

- создать семью;
- отправить заявку;
- отказ скрывает семью;
- 3 active заявки на сервис;
- family_full отменяет остальные заявки;
- принять заявку;
- выдать и подтвердить доступ;
- открыть реквизиты;
- первый платеж paid;
- owner mark not received;
- participant cancel report;
- removal warning;
- family closing.

## Что не делать в первом релизе

- чеки;
- внутренние споры;
- публичный рейтинг;
- админку;
- аккаунты;
- гигабайты;
- сложные тарифные слоты операторов;
- escrow;
- модерацию платежей;
- хранение телефонов пользователей;
- Telegram group creation.
