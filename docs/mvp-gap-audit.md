# Family Engine MVP gap audit

Дата проверки: 2026-06-18.

Этот документ связывает утвержденные продуктовые правила с текущей
реализацией. Он не расширяет MVP аккаунтами, гигабайтами, escrow, рейтингом,
спорами или админкой.

## Автономные пробелы

### P0: обязательные до локальной готовности MVP

- [x] Скрывать из поиска собственные семьи, текущие членства, pending-заявки и
  семьи с постоянным запретом после отказа.
- [x] Открывать username владельца только после активной заявки или вступления
  и давать безопасную кнопку перехода в личный Telegram-чат.
- [x] Отменять будущие `scheduled`-платежи после выхода, удаления или
  окончательного закрытия семьи.
- [x] Не отправлять напоминания по платежам закрытых семей или завершивших
  участие пользователей.
- [x] Покрыть jobs state machine тестами: request expiry, first overdue,
  regular schedule/due/overdue, removal execution и family close.
- [x] Добавить API integration tests для критичного happy path и правил
  доступа к реквизитам.

### P1: утвержденные функции полного Family Engine MVP

- [x] Предоплата одного будущего периода участником.
- [x] Ручное подтверждение владельцем нескольких будущих периодов после
  договоренности сторон.
- [x] Явное отображение отмененных будущих платежей в истории.
- [x] Отдельный сквозной UI-сценарий семьи тарифа, не смешанный с подписками.
- [x] Полные owner/member empty, loading и action states без технических
  сообщений.
- [x] Актуальный audit timeline для всех новых действий.

### P2: hardening для ожидаемой нагрузки

- [x] Проверить индексы частых запросов и ограничения статусов PostgreSQL.
- [x] Добавить anti-abuse ограничения: лимиты заявок, авторизацию реквизитов и
  cooldown ручных напоминаний без дополнительных персональных данных.
- [x] Проверить конкурентное одобрение последнего места на PostgreSQL.
- [x] Устранить расхождения API-контракта, схемы БД и фактических endpoint.
- [x] Добавить production smoke-команды без dev-auth и demo endpoints.

## Внешние блокеры

Эти пункты нельзя завершить локальным кодом без внешнего состояния:

- активация и проверка production-каталога с реальными лимитами сервисов;
- подключение production PostgreSQL/Supabase URL;
- Vercel и Render deployment;
- установка Telegram webhook и Mini App URL;
- проверка внутри реальных Telegram iOS, Android и Desktop клиентов;
- смена ранее опубликованного токена Telegram-бота через BotFather.

## Критерий завершения автономного цикла

- все P0 закрыты;
- локально выполнимые P1 и P2 закрыты либо явно обоснованно перенесены;
- документация совпадает с кодом;
- backend lint/compile/tests, frontend build, Playwright и diff check проходят;
- локальный backend и Mini App подняты на актуальной версии.
