# Доменная модель Family Engine

Этот документ переводит продуктовые правила MVP в инженерную модель. Он не
заменяет [mvp-spec.md](mvp-spec.md), а делает из нее рабочий каркас для кода.

## Модули первого этапа

Первый этап использует только эти модули:

- `identity` - Telegram-вход, обязательный username и минимальный профиль;
- `catalog` - проверенный каталог семейных цифровых сервисов;
- `families` - семьи, заявки, участники, доступ, платежи и закрытие;
- `notifications` - Telegram-уведомления и deep links в Mini App;
- `audit` - история критичных действий Family Engine.

Marketplace, рейтинги, споры и админка не должны попадать в доменную модель
семьи подписки. Продажа гигабайтов и аккаунтов реализована отдельным модулем
`marketplace` и не использует сущности Family Engine.

## Identity

### User

`User` - локальный пользователь SubsMarket, связанный с Telegram.

Поля домена:

- `id`;
- `telegram_user_id`;
- `username`;
- `first_name`;
- `last_name`;
- `photo_url`;
- `created_at`;
- `updated_at`.

Правила:

- `username` обязателен для функциональной части Mini App;
- номер телефона пользователя не запрашивается и не хранится;
- платежный номер владельца не является полем профиля пользователя.

## Catalog

### FamilyService

`FamilyService` - сервис из проверенного каталога семейных подписок.

Поля домена:

- `id`;
- `slug`;
- `name`;
- `category`;
- `max_members`;
- `supported_periods`;
- `status`.

Правила:

- создать семью можно только для сервиса со статусом `active`;
- `max_members` сервиса ограничивает вместимость семьи;
- глобальный максимум MVP - 8 участников с учетом владельца;
- месячная и годовая семьи одного сервиса являются разными `Family`.

## Families

### Family

`Family` - агрегатная корневая сущность Family Engine.

Поля домена:

- `id`;
- `service_id`;
- `owner_user_id`;
- `family_type`;
- `plan_name` - конкретное название плана только для семьи тарифа;
- `status`;
- `period`;
- `max_members`;
- `active_members_count`;
- `total_price_kzt`;
- `member_share_kzt`;
- `rounding_delta_kzt`;
- `payment_day`;
- `next_payment_date`;
- `description`;
- `owner_rules`;
- `created_at`;
- `updated_at`;
- `closing_started_at`;
- `closes_at`.

Статусы:

- `active` - семья принимает заявки, если есть свободные места;
- `full` - семья заполнена и скрыта из поиска;
- `closing` - владелец запустил закрытие, новые заявки не принимаются;
- `closed` - семья закрыта.

Инварианты:

- владелец всегда занимает одно место;
- `active_members_count <= max_members`;
- `max_members <= min(service.max_members, 8)`;
- `member_share_kzt = ceil_to_50(total_price_kzt / max_members)`;
- `rounding_delta_kzt = member_share_kzt * max_members - total_price_kzt`;
- сервис, вместимость, период и правила владельца не меняются после создания;
- описание можно менять без ограничения;
- дату оплаты можно менять только пока семья ни разу не была заполнена;
- цену можно менять не чаще одного раза в календарный месяц и только на будущий
  период;
- заполненная семья скрыта из поиска, но доступна участникам в `Мои семьи`;
- после отказа владельца семья скрывается из поиска для конкретного кандидата.

### FamilyMember

`FamilyMember` - участие пользователя в семье.

Поля домена:

- `id`;
- `family_id`;
- `user_id`;
- `role`;
- `status`;
- `joined_at`;
- `access_provided_at`;
- `access_confirmed_at`;
- `left_at`;
- `removed_at`;
- `removal_scheduled_at`.
- `removal_reason`.

Роли:

- `owner`;
- `member`.

Статусы участника:

- `awaiting_access` - заявка принята, владелец еще не выдал доступ;
- `awaiting_confirmation` - владелец отметил выдачу доступа, участник проверяет;
- `payment_due` - доступ подтвержден, реквизиты открыты, первый платеж ожидается;
- `active` - участник активен;
- `removal_pending` - legacy-статус старого отложенного удаления;
- `left` - участник вышел сам;
- `removed` - участник удален владельцем;
- `cancelled_before_access` - вступление отменено до выдачи доступа.

Правила:

- `FamilyMember` для обычного участника создается только после одобрения заявки;
- пока доступ не выдан, владелец и участник могут отменить вступление без
  удаления и без платежей;
- если отменил владелец, кандидат получает уведомление;
- после отмены до выдачи доступа участник может снова подать заявку в эту же
  семью без штрафа;
- через 24 часа после выдачи доступа система напоминает участнику и владельцу;
- если подтверждения нет, членство не отменяется автоматически и место остается
  занятым;
- владелец может повторно отправить напоминание или немедленно удалить
  участника, обязательно указав причину;
- после удаления место освобождается сразу, а будущие платежи отменяются;
- платеж не создается до подтверждения доступа участником;
- участник может выйти сам в любой момент, место освобождается сразу;
- владелец не выходит как обычный участник, а закрывает семью;
- удаление владельцем выполняется сразу с обязательной причиной;
- причина и действие сохраняются в истории;
- место освобождается сразу, будущие платежи отменяются.

### FamilyRequest

`FamilyRequest` - заявка пользователя в конкретную семью.

Поля домена:

- `id`;
- `family_id`;
- `user_id`;
- `status`;
- `cancel_reason`;
- `created_at`;
- `expires_at`;
- `decided_at`;
- `cancelled_at`.

Статусы:

- `pending`;
- `approved`;
- `rejected`;
- `cancelled`;
- `expired`.

Причины `cancelled`:

- `user_cancelled`;
- `family_full`;
- `owner_cancelled_before_access`;
- `candidate_cancelled_before_access`.

Правила:

- одновременно может существовать только одна `pending`-заявка пользователя в
  одну семью;
- одновременно может существовать максимум три `pending`-заявки пользователя на
  один сервис;
- четвертая активная заявка блокируется текстом:
  `У вас уже 3 активные заявки на {service}. Дождитесь ответа или отмените одну заявку.`;
- владелец видит заявки старые сверху;
- владелец отклоняет заявку одной кнопкой без причины;
- `rejected` финален для пары пользователь-семья;
- после `rejected` повторная заявка в эту семью запрещена, а семья скрывается из
  поиска для кандидата;
- кандидат получает уведомление:
  `Заявка отклонена. Вы можете отправить заявку в другую семью.`;
- кандидат может отменить `pending`-заявку;
- владелец получает уведомление `{name} отменил заявку.`;
- после самостоятельной отмены можно сразу подать повторно, но максимум две
  такие заявки в одну семью;
- `pending` живет 24 часа, затем становится `expired`;
- `expired` не считается отказом и не расходует лимит самостоятельных отмен;
- при `expired` уведомления получают кандидат и владелец;
- если семья заполнилась, остальные `pending` переходят в `cancelled` с причиной
  `family_full`, и кандидат получает уведомление.

### FamilyPayment

`FamilyPayment` - платежное обязательство одного участника за один период.

Поля домена:

- `id`;
- `family_id`;
- `member_id`;
- `period`;
- `kind`;
- `status`;
- `amount_kzt`;
- `due_at`;
- `period_start`;
- `period_end`;
- `requisites_opened_at`;
- `reported_paid_at`;
- `confirmed_paid_at`;
- `overdue_at`;
- `cancelled_at`;
- `cancel_reason`.

Типы:

- `first`;
- `regular`;
- `prepaid`.

Статусы:

- `scheduled`;
- `due`;
- `payment_reported`;
- `paid`;
- `overdue`;
- `cancelled`.

Правила:

- первый платеж создается только после подтверждения доступа участником;
- реквизиты открываются только после подтверждения доступа;
- первый платеж имеет срок 30 минут;
- регулярный платеж становится `overdue` через 24 часа после даты оплаты, если
  участник не нажал `Оплатил`;
- `overdue` не удаляет участника автоматически;
- `Оплатил` переводит платеж в `payment_reported`;
- владелец вручную переводит платеж в `paid`;
- после подтверждения владельцем участник получает `Владелец подтвердил оплату.`;
- если владелец нажал `Не получил оплату`, платеж возвращается в `due`, а
  участник получает `Владелец не подтвердил получение оплаты. Проверьте перевод или свяжитесь с владельцем.`;
- участник может отменить ошибочное `Оплатил` до подтверждения владельцем;
- участник может создать только один ближайший будущий `prepaid` период;
- несколько будущих периодов записывает владелец после договоренности;
- при выходе, удалении или закрытии будущие `scheduled` платежи переходят в
  `cancelled`, неотправленные напоминания отменяются;
- подтвержденный `paid` не меняется задним числом внутри SubsMarket;
- ошибки после подтверждения стороны решают напрямую вне платформы.

### FamilyAuditLog

`FamilyAuditLog` фиксирует критичные события.

Минимальные события:

- создание семьи;
- изменение цены;
- изменение даты оплаты;
- создание, отмена, истечение, отказ и одобрение заявки;
- выдача доступа;
- подтверждение доступа;
- открытие реквизитов;
- отметка `Оплатил`;
- отмена отметки `Оплатил`;
- подтверждение оплаты;
- `Не получил оплату`;
- просрочка;
- выход;
- запуск удаления;
- отмена удаления владельцем;
- удаление;
- запуск закрытия;
- подтверждение предупреждения о закрытии;
- закрытие семьи.

Аудит не принимает бизнес-решения. Он только записывает факты.
